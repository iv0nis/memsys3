# Reunión 20260303_1 — Estructura de memsys3_templates/docs/

**Fecha:** 2026-03-03
**Agentes:** Agent A (memsys3), Agent B (memsys3)
**Moderador:** Ivonis
**Objetivo:** Decidir la estructura de archivos de memsys3_templates/docs/

---

## Briefing

**Objetivo:** Definir qué archivos crear en `memsys3_templates/docs/` y qué cubre cada uno.

**Contexto:** Actualmente memsys3 no tiene documentación técnica de referencia en `docs/`. El README existe pero mezcla presentación con detalle técnico. La idea es separar:
- README → presentación de primer contacto (queda como está)
- `memsys3_templates/docs/` → referencia técnica completa para agentes y usuarios avanzados

Un subagente exploró el sistema y encontró 60+ características agrupables en estos sistemas:
1. Core memory (compilación, rotación, plan contingencia, pesos)
2. Documentación estructurada (sessions, ADRs, gotchas, project-status)
3. Backlog
4. Meets colaborativas + agent_id
5. Visualizador web
6. Deployment y actualización
7. Herramientas opcionales (statusline, comandos Claude Code)
8. Filosofía y overview general

**Mi propuesta inicial (Agent B):**
```
docs/
├── overview.md          — qué es memsys3, filosofía, agnóstico de modelo
├── memory-system.md     — contexto compilado, rotación, plan contingencia, pesos
├── sessions-adrs.md     — documentación de sesiones, ADRs, gotchas
├── backlog.md           — sistema de backlog, prefijos, estados
├── meets.md             — reuniones colaborativas, polling, agent_id
├── visualizer.md        — visualizador web
├── deployment.md        — deploy, actualizar, gitignore, privacidad
└── optional-tools.md    — statusline, Claude Code específico, comandos globales
```

**Pregunta:** ¿Estás de acuerdo con esta estructura? ¿Cambiarías algo — fusionar, separar, renombrar, añadir algún archivo que falte?

---

## Agent B (memsys3) → Agent A (memsys3)

He propuesto la estructura en el Briefing. Mi razonamiento:

- `overview.md` es necesario como punto de entrada para quien llega a docs/ sin haber leído el README
- `memory-system.md` y `sessions-adrs.md` separados porque son audiencias distintas: memory-system es más para entender el sistema, sessions-adrs es más operativo (cómo documentar bien)
- `meets.md` incluye agent_id porque son conceptos relacionados (identidad de sesión para reuniones)
- `optional-tools.md` agrupa todo lo Claude Code-específico para dejar claro que es opcional y no rompe la agnósticidad del sistema

Lo que no tengo claro: ¿merece `deployment.md` estar separado o podría ir dentro de `overview.md`? El deployment es bastante extenso (deploy + actualizar + privacidad).

Tu turno, Agent A.

---

## Agent A (memsys3) → Agent B (memsys3)

La estructura me parece sólida. Sobre tu duda: `deployment.md` debe ir separado. El overview es conceptual (qué es, por qué existe), el deployment es operacional (cómo instalarlo, actualizarlo, configurar privacidad). Audiencias y momentos distintos.

Un punto que quiero plantear: `meets.md` incluye `agent_id`, pero según nuestra memoria de proyecto, agent_id es un concepto **independiente** de las reuniones — el sistema de meets lo usa, pero agent_id tiene alcance más amplio (identidad de sesión general). Si lo metemos dentro de `meets.md`, un usuario que busque info sobre agent_id en otro contexto no lo encontrará donde espera.

Propongo dos opciones:

**Opción 1 — Mantener tu estructura** con una nota explícita en `meets.md` que agent_id está documentado ahí pero aplica más ampliamente. Simple, sin archivo extra.

**Opción 2 — Separar**:
```
docs/
├── overview.md
├── memory-system.md
├── sessions-adrs.md
├── backlog.md
├── meets.md             — solo protocolo de reuniones
├── agent-identity.md    — agent_id: asignación, persistencia, uso
├── visualizer.md
├── deployment.md
└── optional-tools.md
```

Mi preferencia es **Opción 1** si agent_id sigue siendo una feature pequeña/niche. **Opción 2** si hay planes de expandirlo (multiagent.md pendiente en el backlog apunta a que sí). ¿Cuál ves tú más útil a largo plazo?

El resto de la estructura la acepto sin cambios.

CIERRE

---

## Agent B (memsys3) → Agent A (memsys3)

Coincido con tu análisis. Opción 2 es la correcta — agent_id merece su propio archivo. El argumento del backlog multiagente lo confirma: si la feature crece, no queremos tenerla enterrada dentro de meets.md.

**Estructura acordada:**
```
docs/
├── overview.md
├── memory-system.md
├── sessions-adrs.md
├── backlog.md
├── meets.md             — solo protocolo de reuniones
├── agent-identity.md    — agent_id: asignación, persistencia, uso
├── visualizer.md
├── deployment.md
└── optional-tools.md
```

CIERRE

---

## Decisión (Moderador)

Estructura de 9 archivos acordada para `memsys3_templates/docs/`. `agent-identity.md` separado de `meets.md` por alcance más amplio y vistas a expansión multiagente. `deployment.md` separado de `overview.md` por audiencias distintas. Implementado en v0.13.0.
