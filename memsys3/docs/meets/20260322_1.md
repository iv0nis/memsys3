### Edge cases que quiero explorar

**Edge case 1: Archivo custom con mismo nombre que archivo nuevo del template**

Escenario: el usuario crea `memsys3/prompts/migrate.md` custom. En la próxima versión del template, memsys3 añade un `migrate.md` oficial. ¿Qué pasa?

Con git diff (estrategia principal): `migrate.md` aparecería como A (añadido). El agente haría `cp` y sobrescribiría el custom del usuario.

Con el fallback dinámico: el loop copia todo lo del template → misma sobrescritura.

**Esto es un bug potencial.** El Paso 5.5 detectaría `migrate.md` como "EN TEMPLATE" (porque ahora sí existe en el template nuevo), cuando antes era CUSTOM.

**Propuesta para este edge case:** En el Paso 5.5, la comparación debería ser contra el template de la **versión actual instalada**, no solo contra el template nuevo. Así:

```
1. Listar archivos en memsys3/ del proyecto
2. Listar archivos en template NUEVO (memsys3_update_temp)
3. Listar archivos en template VIEJO (la versión actual → se puede reconstruir con git checkout $CURRENT_VERSION en el clone)
4. Si un archivo existe en el proyecto pero NO en el template viejo → es CUSTOM
   (independientemente de si existe en el template nuevo)
5. Si es CUSTOM y también existe en template nuevo → CONFLICTO → avisar al usuario
```

Esto es exactamente el three-way diff que mencionaste, pero aplicado a nivel de existencia de archivos, no de contenido.

**Edge case 2: El usuario renombra un archivo del template**

Ej: renombra `endSession.md` → `endSession.original.md` y crea su propio `endSession.md`. Esto no es detectable automáticamente — pero tampoco creo que sea un escenario realista. ¿Lo ves posible?
